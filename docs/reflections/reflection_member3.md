## My Contribution to the Project
For this group project, my assigned responsibility was the Network Configuration Automation component of the automation suite. Specifically, I was tasked with developing the Ansible playbook located at ansible/network/configure_device.yml, which forms part of the broader network and systems automation solution built using Docker, Ansible, and NETCONF. My playbook was designed to automatically push configuration changes to a live Cisco IOS XR network device via NETCONF, fulfilling the project requirement to configure a login banner, a local user account, an interface description, and an IP address on a network interface.

To build this, I used Ansible's netconf_config module to send YANG-modelled XML payloads to the Cisco DevNet IOS XR Always-On sandbox. The banner task used the Cisco-IOS-XR-infra-infra-cfg YANG model with the correct delimiter format required by IOS XR. The interface description and IP address tasks used the Cisco-IOS-XR-ifmgr-cfg and Cisco-IOS-XR-ipv4-io-cfg namespaces respectively, targeting GigabitEthernet0/0/0/1. The user account task was structured using the Cisco-IOS-XR-aaa-locald-cfg model, though it was set to ignore_errors: yes since the shared sandbox restricts AAA modifications for security reasons. I also set up the inventory file at ansible/network/inventory.yml with the sandbox host details and credentials. Beyond my assigned scope, I assisted in resolving the static route configuration issue by identifying the correct YANG element path that the sandbox accepted.

Before the full Docker environment was ready, I tested the playbook directly from my Oracle VM terminal using ansible-playbook -i ansible/network/inventory.yml ansible/network/configure_device.yml, connecting directly to the live DevNet sandbox over NETCONF on port 830. This allowed me to validate each task individually and confirm that configurations were being applied to the device. Once I confirmed the playbook worked correctly, I committed and pushed my changes to the shared GitHub repository.

## Challenges Encountered
The most recurring challenge was managing the DevNet sandbox credentials. The IOS XR Always-On sandbox generates unique per-session credentials that expire periodically, requiring me to fetch a new username and password from the DevNet portal each session and update inventory.yml accordingly. Early on, I was unaware of this per-session system and spent significant time troubleshooting authentication failures before understanding the root cause. I also encountered SSH known_hosts conflicts whenever the sandbox rotated its host key, which required running ssh-keygen -R to clear the cached entry before reconnecting.

Another significant challenge was an Ansible version conflict on the VM. The system had Ansible 2.9.6 installed at the system level, which did not support the ansible.netcommon collection required by the original playbook. Upgrading Ansible via pip installed a newer version at a different path, but the system continued defaulting to the old version due to PATH priority. After troubleshooting, I resolved this by rewriting the playbook to use the built-in netconf_config module compatible with Ansible 2.9, which was already available without any additional collection. Getting the YANG XML syntax correct also required several iterations, particularly the banner delimiter format and understanding the correct element names for the static route configuration.

## What I Learned From the Project
Technically, I learned how to write Ansible playbooks for network device automation using NETCONF and YANG data models, which I had only studied theoretically before. I gained a practical understanding of how YANG namespaces work in IOS XR, how the candidate datastore differs from the running datastore, and why certain configurations are restricted in shared sandbox environments. I also learned the importance of testing NETCONF connectivity independently before running a full playbook, using raw SSH as a quick way to verify credentials and sandbox availability.

On the collaboration side, working in a shared repository taught me the importance of pulling before editing, committing clearly, and communicating blockers to teammates early. Overall, this project gave me hands-on confidence in network programmability and automation that goes well beyond what classroom exercises alone could provide.
