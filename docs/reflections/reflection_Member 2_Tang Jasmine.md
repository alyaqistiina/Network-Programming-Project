# Personal Reflection - Member 2 (Tang Jasmine, A23CS0277)
Role: Linux System Information Automation (Ansible Playbook Developer)

## My Contribution to the Project
For this group project, my assigned responsibility was the Linux System Information Automation component of the automation suite. Specifically, I was tasked with developing the Ansible playbook located at ansible/linux/system_info.yml, which forms part of the broader network and systems automation solution built using Docker, Ansible, and NETCONF. My playbook was designed to automatically collect and display key system information from a Linux target, fulfilling the project requirement to gather: hostname, current date and time, CPU information, memory usage, disk usage, logged-in users, and the top five running processes by CPU usage.

To build this, I used Ansible's command and shell modules to execute standard Linux diagnostic commands, including uname -a for hostname and kernel details, date for the current date and time, lscpu for CPU specifications, free -h for memory usage in human-readable format, df -h for disk usage, who for currently logged-in users, and ps aux --sort=-%cpu | head -6 to identify the five most CPU-intensive processes. Each command's output was captured using the register keyword and then displayed cleanly to the terminal using Ansible's debug module.I set gather_facts: false at the play level, since the playbook relies on explicit shell commands rather than Ansible's built-in fact-gathering, which reduced execution time. I also set changed_when: false on every task, since these are all read-only, information-gathering operations that do not modify system state. Tthis ensured Ansible reported an accurate "ok" status rather than a misleading "changed" status.

Before integrating with the team's shared Linux target container, I tested the playbook locally on my own Oracle VM environment using the command ansible-playbook -i localhost, -c local ansible/linux/system_info.yml. This allowed me to validate that each task executed correctly and produced the expected output without requiring the full Docker/SSH infrastructure to be in place yet. Once I confirmed the playbook worked as intended, I committed my changes and pushed them to our shared GitHub repository so the rest of the team could review and later integrate it with the Linux target container defined in the Docker Compose setup.

## Challenges Encountered
One challenge was discovering that my commits were being attributed to “Normizan”  instead of mine. This happened because the repository had a local Git configuration that overrode my global settings. Git configuration works in layers which includes system, global, and local, and local settings always take priority. I fixed this by setting git config --local user.name and user.email for the repository, and used git commit --amend --reset-author to correct an existing commit before pushing. This taught me to always check Git identity settings in a shared environment rather than assuming my global config applies. Another challenge was working without the full integrated environment. I conduct the testing locally first with -i localhost, -c local, which lets me validate my playbook's logic independently before integration. It is  a useful lesson in building and testing components in isolation. Finally, I learnt the playbook structure, the difference between command and shell, and how variable registration works with the debug module. Small early errors, like confusing stdout with stdout_lines, took some trial and error but helped solidify my understanding.

## What I Learned From the Project
Technically, I learned how to write a working Ansible playbook from scratch, structuring tasks, registering command output, and displaying it cleanly. I also learned why changed_when: false matters for read-only tasks, and gained a better sense of how a component like this fits into a larger automation architecture involving Docker and SSH. On the collaboration side, I learned that Git in a team setting takes more than just add, commit, and push. Understanding configuration layers, rebasing before pushing, and fixing authorship issues are skills that only really surface when working in a shared repository. Having a dedicated file for my component also made it much easier to work independently without disrupting my teammates' progress. I also learned the value of testing a component locally before full integration, rather than waiting for every piece of infrastructure to be ready. This is a habit I plan to carry into future projects, where complete environments are rarely available from the start. Overall, this project strengthened both my technical skills in Ansible automation and my practical understanding of collaborative development using Git and GitHub.
